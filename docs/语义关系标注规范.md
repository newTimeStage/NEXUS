# NEXUS 语义关系标注规范

## 1. 文档定位

本文定义 `NEXUS` 项目的《语义关系标注规范》。

目标不是做一份抽象概念说明，而是提供一套可以直接用于：

1. 人工标注
2. 编译期抽取
3. 关系校验
4. 索引生成
5. 后续知识图谱构建
6. AI Agent 辅助标注

的统一标准。

本规范基于三类现实约束制定：

1. 基于已验证的经验：先做稳定的语义标注与静态知识资产，再做图谱应用
2. `NEXUS` 当前现状：已经具备实体标注、段落编号、编译期 Markdown 处理能力，但尚未建立关系层标准
3. 技术路线方案：遵循 Phase 4（关系图谱与质量校验）的设计思路，先做规则关系，再做推理关系

---

## 2. 设计原则

### 2.1 编译期优先

关系标注必须优先服务于“离线提取”和“静态构建”，而不是依赖运行时推理。

### 2.2 正文轻量、关系外置

正文中的实体标注继续保持轻量；关系标注原则上采用独立关系块或伴随文件承载，避免污染长文阅读体验。

### 2.3 一条关系只表达一个命题

一个关系记录只能表达一个明确命题，不能在一条记录中同时塞入多个动作或多个逻辑判断。

### 2.4 规范名优先

关系两端必须使用规范名或稳定 ID，不使用仅在局部可理解的简称。

### 2.5 证据必填

任何关系如果没有证据锚点，就不能进入正式关系层。

### 2.6 显式区分“明示”“推断”“争议”“否定”

关系不是只有“有”或“没有”。必须把关系状态单独标出，避免把推断结果伪装成原文明示事实。

---

## 3. 适用范围

本规范适用于 `NEXUS` 中以下内容：

- 历史叙事
- 制度演进
- 思想传承
- 文献互引
- 人物、组织、事件之间的结构关系

本规范暂不覆盖：

- 纯视觉布局关系
- HTML/CSS 展示关系
- 自动生成但未经人工确认的实验性边

---

## 4. 核心术语

### 4.1 实体 Entity

已在正文中以 `【type:内容】` 标注的对象，如人名、地名、制度、典籍、思想等。

### 4.2 事件 Event

在叙事中可被识别为“发生了什么”的单元，通常带有时间、参与者、地点或结果。

### 4.3 关系 Relation

连接两个节点的有方向或无方向命题。

### 4.4 节点 Node

关系两端对象的统称，可为：

- 实体
- 事件
- 段落
- 小节
- 文档

### 4.5 证据 Evidence

支撑该关系成立的原文位置，至少应精确到段落编号。

### 4.6 规范名 Canonical Name

用于计算、聚合、索引的统一命名。简称、别名、雅号都应回收到规范名。

## 5. 关系标注载体

## 5.1 推荐载体

推荐使用独立 Markdown 伴随文件承载关系标注：

```text
原文:     src/content/.../文章名.md
关系文件: src/content/.../文章名.relations.json
```

原因：

- 不干扰正文阅读
- 便于批量抽取
- 便于差异审查
- 便于后续把正文和关系层分离构建

## 5.2 允许的备选载体

如确需与正文同文件维护，可在文末放置一个独立关系块，但不得插入正文段落中间。

## 5.3 统一语法

无论是伴随文件还是文末附录，都使用同一种 fenced block：

````markdown
```nexus-relations
- id: rel-001
  type: parent_of
  source: entity:person:周文王
  target: entity:person:周武王
  evidence:
    - para:12
  status: asserted
  confidence: high
  note: 原文明确记其父子关系
```
````

说明：

- `nexus-relations` 为唯一合法关系块类型
- 块内数据格式采用 YAML 列表
- 一个 block 内可包含多条关系

## 6. 节点引用语法

关系两端必须使用显式节点引用，不允许裸文本。

统一格式：

```text
<node-kind>:<subtype>:<canonical-name-or-id>
```

## 6.1 实体引用

```text
entity:person:孔子
entity:place:长安
entity:institution:郡县制
entity:book:尚书
```

要求：

- `subtype` 必须与实体类型一致
- 名称使用规范名

## 6.2 事件引用

```text
event:war:牧野之战
event:politics:王莽改制
event:id:evt-000123
```

要求：

- 若已有稳定事件 ID，优先使用 `event:id:*`
- 若暂无事件库，可先用 `event:<type>:<canonical-title>`

## 6.3 段落引用

```text
para:1
para:2.1
para:12.3
```

段落编号必须与正文中的 Purple Number 锚点一致。

## 6.4 小节引用

```text
section:黄帝
section:礼乐制度
```

仅用于结构关系，不作为首选证据锚点。

## 6.5 文档引用

```text
doc:制度与创造/政典之书/01-中国共产党政典
```

用于跨文档关系。

## 7. 关系记录字段规范

每条关系记录字段如下。

## 7.1 必填字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 关系 ID，文件内唯一 |
| `type` | enum | 关系类型，必须在白名单中 |
| `source` | string | 源节点引用 |
| `target` | string | 目标节点引用 |
| `evidence` | list | 证据锚点列表，至少一个 |
| `status` | enum | 关系状态 |
| `confidence` | enum | 关系置信度 |

## 7.2 可选字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `note` | string | 补充说明 |
| `source_doc` | string | 当 `source` 不在当前文档时填写 |
| `target_doc` | string | 当 `target` 不在当前文档时填写 |
| `rules` | list | 触发该关系的规则名或推理依据 |
| `tags` | list | 分类标签 |

## 7.3 `status` 取值

| 值 | 含义 |
|------|------|
| `asserted` | 原文直接明示 |
| `inferred` | 依据原文可作单步或有限推断 |
| `disputed` | 文内或跨文存在冲突记载 |
| `negated` | 原文明示该关系不成立 |

## 7.4 `confidence` 取值

| 值 | 含义 |
|------|------|
| `high` | 原文明确或证据强 |
| `medium` | 证据较充分，但仍有解释空间 |
| `low` | 暂定关系，待复核 |

## 8. 关系类型白名单

为避免关系类型失控，`v1.0` 只允许使用下列白名单关系。

## 8.1 身份与指称关系

| 类型 | 含义 | 方向 | 对称性 |
|------|------|------|------|
| `same_as` | 两节点为同一对象 | 双向 | 是 |
| `alias_of` | 源节点是目标节点的别名/简称 | alias -> canonical | 否 |
| `refers_to` | 文本单元指向某实体/事件 | source -> target | 否 |
| `cites` | 文本/典籍引用另一文本/典籍 | citing -> cited | 否 |
| `interprets` | 文本/人物对对象进行解释 | interpreter -> target | 否 |

## 8.2 结构与论述关系

| 类型 | 含义 | 方向 | 对称性 |
|------|------|------|------|
| `part_of` | 源节点是目标节点的组成部分 | part -> whole | 否 |
| `contains` | 源节点包含目标节点 | whole -> part | 否 |
| `elaborates` | 源节点展开说明目标节点 | elaboration -> topic | 否 |
| `supports` | 源节点支持目标命题 | support -> claim | 否 |
| `refutes` | 源节点反驳目标命题 | refutation -> claim | 否 |
| `contrasts_with` | 两节点构成对照 | weaker canonical order | 是 |

规范要求：

- `part_of` 与 `contains` 二选一，推荐统一使用 `part_of`
- `contrasts_with` 为对称关系，避免同时写两条镜像记录

## 8.3 时间与因果关系

| 类型 | 含义 | 方向 | 对称性 |
|------|------|------|------|
| `precedes` | 源节点发生在目标之前 | earlier -> later | 否 |
| `concurrent_with` | 两节点同期/并行 | 双向 | 是 |
| `causes` | 源节点导致目标节点 | cause -> effect | 否 |
| `results_in` | 源节点造成目标结果 | action -> result | 否 |
| `motivates` | 源节点构成目标动因 | motive -> action | 否 |

规范要求：

- `precedes` 为时间标准方向，禁止再定义 `follows`
- `causes` 与 `results_in` 不可混用在同一命题上，优先选更自然的一种

## 8.4 人物与社会关系

| 类型 | 含义 | 方向 | 对称性 |
|------|------|------|------|
| `parent_of` | 父母/上代到子代 | parent -> child | 否 |
| `spouse_of` | 婚配关系 | 双向 | 是 |
| `sibling_of` | 兄弟姊妹关系 | 双向 | 是 |
| `teacher_of` | 师承关系 | teacher -> student | 否 |
| `serves_under` | 臣属/部属关系 | subordinate -> superior | 否 |
| `appoints` | 任命关系 | appointing side -> appointee | 否 |
| `succeeds` | 继任关系 | successor -> predecessor | 否 |
| `opposes` | 对立关系 | source -> target | 可成对 |
| `allies_with` | 联盟/同盟关系 | 双向 | 是 |

规范要求：

- 家庭关系优先使用最稳定的标准谓词，不用自由发挥
- 若原文只明示“子”，但未明示父或母，可仍使用 `parent_of`，并在 `note` 中补明

## 8.5 事件参与与制度关系

| 类型 | 含义 | 方向 | 对称性 |
|------|------|------|------|
| `participates_in` | 实体参与事件 | entity -> event | 否 |
| `occurs_in` | 事件发生于地点/空间 | event -> place | 否 |
| `occurs_on` | 事件发生于时间点/时间段 | event -> time | 否 |
| `governs` | 人物/政体治理某对象 | ruler -> target | 否 |
| `member_of` | 个体隶属某组织/群体 | member -> group | 否 |
| `establishes` | 创建制度/组织/政权 | founder -> created object | 否 |
| `abolishes` | 废除制度/组织/旧秩序 | actor -> abolished object | 否 |
| `influences` | 对思想/制度/人物产生影响 | source -> target | 否 |

## 9. 规范化规则

## 9.1 方向统一规则

同一关系类型必须始终使用同一方向。例如：

- `parent_of` 只能是“父母 -> 子女”
- `teacher_of` 只能是“老师 -> 学生”
- `serves_under` 只能是“臣属 -> 君主/上级”
- `precedes` 只能是“早 -> 晚”
- `causes` 只能是“原因 -> 结果”
- `part_of` 只能是“部分 -> 整体”
- `establishes` 只能是“创建者 -> 创建对象”

## 9.2 对称关系去重规则

以下关系只保留一条：

- `same_as`
- `spouse_of`
- `sibling_of`
- `allies_with`
- `concurrent_with`
- `contrasts_with`
- `same_paragraph`
- `co_person`
- `co_place`
- `cross_ref`

如需稳定排序，按规范名字典序较小者放 `source`。

## 9.3 规范名规则

关系两端不得写：

- 仅在上下文中才成立的代词
- 未消歧简称
- 非稳定临时标签

例如：

- 错误：`entity:person:武王`
- 正确：`entity:person:周武王`

与实体索引中的 canonical name 保持一致，遵循规范名与别名消歧规范。

## 9.4 证据锚点规则

`evidence` 至少包含一个 `para:*`，推荐写法：

```yaml
evidence:
  - para:12
  - para:12.1
```

跨文档关系需额外填写：

```yaml
source_doc: doc:演进轨迹/...
target_doc: doc:制度与创造/...
```

证据锚点必须与正文中的 Purple Number 锚点一致，确保可验证性。

## 9.5 一条关系只写一个谓词

错误示例：

```yaml
- id: rel-001
  type: causes_and_precedes
```

正确做法：拆成两条。

## 9.6 AI Agent 标注规范

AI Agent 生成的关系标注应：

- 使用 `status: ai_generated` 标记
- 使用 `confidence: ai_estimated` 标记
- 提供详细的 `note` 说明推理过程
- 包含 `rules` 字段说明触发规则
- 由人工审核后转为正式状态
- 遵循与人工标注相同的格式和规范

## 9.7 与技术路线方案的衔接

- 规则关系应与 Phase 4 中的规则关系生成脚本对接
- 推理关系应与 Phase 4 中的质量校验机制对接
- 所有关系应支持 Phase 5 中的图谱可视化需求
- 关系数据结构应兼容 Phase 6 中的本体/RDF/图数据库扩展

## 9.8 质量控制机制

建立质量控制闭环：

1. **抽取**：从标注文本中提取关系
2. **校验**：使用脚本验证关系的合法性
3. **反思**：人工审核和修正
4. **批改**：更新关系标注
5. **重建**：重新生成关系索引和图谱

这个闭环是知识图谱质量逐步提升的关键。

## 12. 禁止写法

以下写法一律禁止。

## 12.1 使用未定义关系类型

错误：

```yaml
type: 非常重要
```

## 12.2 使用裸文本节点

错误：

```yaml
source: 武王
target: 商
```

## 12.3 没有证据

错误：

```yaml
- id: rel-010
  type: influences
  source: entity:concept:法家
  target: entity:institution:秦制
```

## 12.4 把推断关系伪装成明示关系

如果原文没有直说，就不能把 `status` 写成 `asserted`。

## 12.5 在一条关系里混入多个判断

错误：

```yaml
- id: rel-011
  type: causes
  source: event:war:牧野之战
  target: event:politics:商亡与周兴
```

正确做法：拆分成两条关系。

## 13. 标注流程建议

### 13.1 人工标注流程

推荐按以下顺序工作：

1. **实体标注**：完成正文中的实体标注与段落编号，标注于段尾，不能影响段落整体的Markdown语法
2. **规范名确认**：确认实体的规范名，处理别名和消歧
3. **关系标注**：补写关系文件，建立实体、事件之间的关系
4. **脚本校验**：运行校验脚本验证关系的合法性
5. **质量审查**：人工审核校验结果，修正错误
6. **重建索引**：重新生成关系索引和图谱数据

不要反过来先写关系，再补实体，否则后续会频繁返工。

### 13.2 AI Agent 辅助标注流程

1. **预处理**：确保正文已完成实体标注和段落编号
2. **AI 生成**：使用 AI Agent 分析文本，生成关系标注建议
3. **人工审核**：审核 AI 生成的关系，修正错误和补充信息
4. **状态转换**：将 `ai_generated` 状态的关系转换为正式状态
5. **脚本校验**：运行校验脚本验证所有关系
6. **质量控制**：进入质量控制闭环

### 13.3 与技术路线方案的衔接

- **Phase 1**：完成统一标注协议
- **Phase 2**：生成实体索引，为关系标注提供基础
- **Phase 3**：完成事件抽取，为事件相关关系提供支持
- **Phase 4**：实现规则关系生成和推理关系审查
- **Phase 5**：基于关系数据构建可视化页面
- **Phase 6**：扩展到本体/RDF/图数据库

## 14. 校验规则

未来脚本应至少检查以下项目：

1. `id` 在当前关系文件内唯一
2. `type` 在白名单中
3. `source` 与 `target` 语法合法
4. `source` 与 `target` 不可相同，除 `same_as` 外
5. `evidence` 非空
6. `status` 与 `confidence` 取值合法
7. 对称关系不存在镜像重复记录
8. 规范名能够在实体索引或事件索引中解析
9. `para:*` 在正文中真实存在
10. 跨文档关系如果不在当前文件，必须填写 `source_doc` 或 `target_doc`

## 15. 与现有实体规范的衔接

本规范依赖现有实体标注规范：

- `【person:孔子】`
- `【place:长安】`
- `【book:尚书】`
- `【concept:仁义】`

关系层不取代实体层，而是建立在实体层之上。

简单说：

- 实体标注解决“这里出现了谁/什么”
- 关系标注解决“这些对象之间是什么关系”

## 17. 最终要求

`NEXUS` 的语义关系标注，必须满足四个底线：

1. **可读**：人能看懂
2. **可抽取**：脚本能稳定解析
3. **可校验**：错误能自动发现
4. **可演进**：将来可生成索引、时间线、关系网、图谱数据

如果某种写法只满足“看起来方便”，但不满足这四点，就不应进入正式规范。
