# NEXUS 规范名与别名消歧规范

## 1. 文档定位

本文定义 `NEXUS` 项目的《规范名与别名消歧规范》。

目标不是做一份抽象概念说明，而是提供一套可以直接用于：

1. 实体标注中的规范名命名
2. 别名与简称的处理
3. 同名实体的消歧
4. 编译期实体索引的生成
5. 关系标注中的实体引用
6. AI Agent 辅助标注

的统一标准。

本规范基于三类现实约束制定：

1. 根据已验证的经验：先做稳定的语义标注与静态知识资产，再做图谱应用
2. `NEXUS` 当前现状：已经具备实体标注、段落编号、编译期 Markdown 处理能力
3. 技术路线方案：遵循 Phase 2（编译期实体索引）的设计思路

---

## 2. 核心概念

### 2.1 规范名（Canonical Name）

用于计算、聚合、索引的统一命名。是实体在系统中的唯一标识，用于知识图谱的节点标识和关系连接。

### 2.2 别名（Alias）

实体的其他名称，包括简称、雅号、别称等。别名应映射到规范名，确保知识的一致性。

### 2.3 消歧（Disambiguation）

解决同名异人、同名异地等问题的过程，确保每个实体都有唯一的规范名。

### 2.4 实体索引（Entity Index）

编译期生成的实体信息集合，包含规范名、别名、类型、出现位置等信息。

---

## 3. 规范名命名规则

### 3.1 基本原则

1. **唯一性**：规范名在同一实体类型内必须唯一
2. **稳定性**：规范名一旦确定，不应轻易更改
3. **可读性**：规范名应便于人类理解和使用
4. **一致性**：相同实体在不同文档中应使用相同的规范名

### 3.2 具体规则

#### 3.2.1 人名

- 使用全名，避免简称
- 历史人物使用通用称谓
- 避免使用含糊不清的称呼

**示例**：
- 正确：`周武王`、`孔子`、`秦始皇`
- 错误：`武王`、`孔丘`、`嬴政`（除非有特殊需要）

#### 3.2.2 地名

- 使用标准地名
- 历史地名使用当时的正式名称
- 避免使用方言或俗称

**示例**：
- 正确：`长安`、`洛阳`、`牧野`
- 错误：`西安`（现代名称）、`洛邑`（古称，除非有特殊需要）

#### 3.2.3 制度

- 使用完整名称
- 避免使用缩写
- 保持术语的一致性

**示例**：
- 正确：`郡县制`、`科举制`、`礼乐制度`
- 错误：`郡县`、`科举`、`礼乐`

#### 3.2.4 典籍

- 使用标准书名
- 包含必要的书名号（在显示时）
- 避免使用简称

**示例**：
- 正确：`论语`、`尚书`、`春秋`
- 错误：`论`、`书`、`春秋经`（除非有特殊需要）

#### 3.2.5 思想概念

- 使用标准术语
- 保持概念的完整性
- 避免使用含糊不清的表达

**示例**：
- 正确：`仁义`、`礼治`、`法家思想`
- 错误：`仁`、`礼`、`法家`

## 4. 别名管理

### 4.1 别名类型

1. **简称**：如 `始皇` → `秦始皇`
2. **雅号**：如 `太白` → `李白`
3. **别称**：如 `孔夫子` → `孔子`
4. **异称**：如 `汉高祖` → `刘邦`

### 4.2 别名映射

使用 `entity-aliases.json` 文件管理别名映射：

```json
{
  "秦始皇": ["始皇", "嬴政", "祖龙"],
  "孔子": ["孔丘", "孔夫子", "仲尼"],
  "长安": ["西京", "京兆"]
}
```

### 4.3 实体标注中的消歧语法

在实体标注中使用消歧语法：

```markdown
【person:始皇|秦始皇】
【place:西京|长安】
```

显示文本是“始皇”，语义归一到“秦始皇”。

## 5. 同名实体消歧

### 5.1 消歧策略

1. **上下文消歧**：根据上下文确定实体的具体所指
2. **地域消歧**：通过地理位置区分同名地点
3. **时间消歧**：通过时间时期区分同名人物
4. **领域消歧**：通过专业领域区分同名概念

### 5.2 消歧映射

使用 `disambiguation-map.json` 文件管理同名实体的消歧：

```json
{
  "文帝": [
    {
      "canonical": "汉文帝",
      "context": "西汉",
      "time_period": "公元前179年-公元前157年"
    },
    {
      "canonical": "魏文帝",
      "context": "三国",
      "time_period": "公元220年-公元226年"
    }
  ]
}
```

### 5.3 消歧标注

在实体标注中使用消歧语法明确指定规范名：

```markdown
【person:文帝|汉文帝】
【person:文帝|魏文帝】
```

## 6. 实体索引结构

### 6.1 实体索引 JSON 结构

```json
{
  "entities": [
    {
      "id": "ent-0001",
      "type": "person",
      "name": "周武王",
      "aliases": ["武王", "姬发"],
      "occurrences": [
        {
          "article": "演进轨迹/周本纪",
          "paragraphs": ["1", "1.1"]
        }
      ]
    }
  ]
}
```

### 6.2 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 实体 ID，全局唯一 |
| `type` | string | 实体类型 |
| `name` | string | 规范名 |
| `aliases` | array | 别名列表 |
| `occurrences` | array | 出现位置列表 |

## 7. 与其他规范的衔接

### 7.1 与实体标注的衔接

- 实体标注中的规范名应与本规范保持一致
- 消歧语法应遵循本规范的规定
- 实体类型应与 `NEXUS` 实体标注系统中的18类实体对应

### 7.2 与事件标注的衔接

- 事件标注中的人物和地点应使用规范名
- 事件时间应与时间实体的规范表达保持一致

### 7.3 与关系标注的衔接

- 关系标注中的节点引用应使用规范名
- 关系两端的实体应在实体索引中存在

## 8. AI Agent 辅助消歧

### 8.1 AI Agent 消歧策略

1. **上下文分析**：分析实体出现的上下文，确定最可能的规范名
2. **知识库匹配**：与现有实体索引匹配，寻找最相似的规范名
3. **规则应用**：应用消歧规则，处理常见的同名异实情况
4. **置信度评估**：为消歧结果提供置信度评估

### 8.2 AI Agent 输出格式

AI Agent 生成的消歧建议应包含：

- 原始文本
- 候选规范名
- 置信度
- 消歧依据

## 9. 质量控制机制

### 9.1 校验规则

1. **唯一性检查**：确保规范名在同一类型内唯一
2. **一致性检查**：确保同一实体在不同文档中使用相同的规范名
3. **完整性检查**：确保所有实体都有规范名
4. **消歧检查**：确保同名实体已正确消歧

### 9.2 质量控制闭环

1. **抽取**：从标注文本中提取实体和别名
2. **校验**：使用脚本验证规范名和别名的合法性
3. **反思**：人工审核和修正消歧结果
4. **批改**：更新实体索引和消歧映射
5. **重建**：重新生成实体索引和相关数据

## 10. 与技术路线方案的衔接

### 10.1 Phase 2 编译期实体索引

- 规范名是实体索引的核心标识
- 别名映射是实体索引的重要组成部分
- 消歧机制确保实体索引的准确性

### 10.2 Phase 3 事件抽取与时间轴

- 事件涉及的实体使用规范名，确保与实体索引一致
- 时间表达的规范确保事件时间线的准确性

### 10.3 Phase 4 关系图谱与质量校验

- 关系两端的实体使用规范名，确保关系的准确性
- 消歧机制避免关系指向错误的实体

### 10.4 Phase 5 图谱可视化页面

- 规范名作为实体节点的显示名称
- 别名作为实体节点的补充信息

## 11. 标注流程建议

### 11.1 人工标注流程

1. **实体标注**：在正文中标注实体，使用消歧语法处理别名和同名实体
2. **规范名确认**：确认每个实体的规范名，确保唯一性和一致性
3. **别名收集**：收集实体的别名，更新别名映射
4. **消歧处理**：处理同名实体，更新消歧映射
5. **脚本校验**：运行校验脚本验证规范名和别名的合法性
6. **重建索引**：重新生成实体索引

### 11.2 AI Agent 辅助标注流程

1. **预处理**：分析文本，识别潜在实体
2. **AI 消歧**：使用 AI Agent 进行实体消歧，生成规范名建议
3. **人工审核**：审核 AI 生成的规范名和别名，修正错误
4. **脚本校验**：运行校验脚本验证所有实体
5. **质量控制**：进入质量控制闭环

## 14. 最终要求

`NEXUS` 的规范名与别名消歧，必须满足四个底线：

1. **唯一性**：规范名在同一类型内唯一
2. **一致性**：同一实体在不同文档中使用相同的规范名
3. **可扩展性**：支持新实体和新别名的添加
4. **可验证性**：消歧结果可通过脚本验证

如果某种命名方式只满足“看起来方便”，但不满足这四点，就不应进入正式规范。
