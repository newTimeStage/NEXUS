# NEXUS 事件标注规范

## 1. 文档定位

本文定义 `NEXUS` 项目的《事件标注规范》。

目标不是做一份抽象概念说明，而是提供一套可以直接用于：

1. 人工标注
2. 编译期抽取
3. 事件索引生成
4. 时间线构建
5. 关系图谱构建
6. AI Agent 辅助标注

的统一标准。

本规范基于三类现实约束制定：

1. 基于已验证的经验：先做稳定的语义标注与静态知识资产，再做图谱应用
2. `NEXUS` 当前现状：已经具备实体标注、段落编号、编译期 Markdown 处理能力
3. 技术路线方案：遵循 Phase 3（事件抽取与时间轴）的设计思路

---

## 2. 设计原则

### 2.1 编译期优先

事件标注必须优先服务于“离线提取”和“静态构建”，而不是依赖运行时推理。

### 2.2 正文轻量、事件外置

正文中的实体标注继续保持轻量；事件标注原则上采用独立事件块或伴随文件承载，避免污染长文阅读体验。

### 2.3 最小可行事件模型

事件标注应包含最核心的信息，避免过度复杂的结构，确保可扩展性。

### 2.4 规范名优先

事件涉及的实体必须使用规范名，与实体索引中的 canonical name 保持一致。

### 2.5 证据必填

任何事件如果没有证据锚点，就不能进入正式事件层。

### 2.6 AI Agent 友好

标注格式和流程设计应考虑 AI Agent 的辅助标注能力，确保结构清晰、规则明确，便于自动化处理和质量控制。

## 3. 事件模型

### 3.1 最小可行事件模型

事件 JSON 结构如下：

```json
{
  "id": "evt-0001",
  "title": "武王伐纣",
  "type": "war",
  "article": "演进轨迹/周本纪",
  "paragraphs": ["1", "1.1"],
  "time_text": "上元6691年",
  "year": 6691,
  "people": ["周武王", "纣王"],
  "places": ["牧野"],
  "summary": "周武王率诸侯伐纣并获胜"
}
```

### 3.2 核心字段说明

| 字段 | 类型 | 说明 | 必要性 |
|------|------|------|--------|
| `id` | string | 事件 ID，全局唯一 | 必填 |
| `title` | string | 事件标题 | 必填 |
| `type` | string | 事件类型 | 必填 |
| `article` | string | 事件所在文章路径 | 必填 |
| `paragraphs` | array | 事件相关段落编号 | 必填 |
| `time_text` | string | 事件时间文本 | 必填 |
| `year` | number | 事件年份（上元纪年） | 必填 |
| `people` | array | 事件涉及人物（规范名） | 可选 |
| `places` | array | 事件涉及地点（规范名） | 可选 |
| `summary` | string | 事件摘要 | 可选 |

## 4. 事件标注载体

### 4.1 推荐载体

推荐使用独立 Markdown 伴随文件承载事件标注：

```text
原文:     src/content/.../文章名.md
事件文件: src/content/.../文章名.events.md
```

原因：

- 不干扰正文阅读
- 便于批量抽取
- 便于差异审查
- 便于后续把正文和事件层分离构建
- 便于 AI Agent 批量处理和验证

### 4.2 允许的备选载体

如确需与正文同文件维护，可在文末放置一个独立事件块，但不得插入正文段落中间。

### 4.3 统一语法

无论是伴随文件还是文末附录，都使用同一种 fenced block：

````markdown
```nexus-events
- id: evt-0001
  title: 武王伐纣
  type: war
  article: 演进轨迹/周本纪
  paragraphs: ["1", "1.1"]
  time_text: "上元6691年"
  year: 6691
  people: ["周武王", "纣王"]
  places: ["牧野"]
  summary: "周武王率诸侯伐纣并获胜"
```
````

说明：

- `nexus-events` 为唯一合法事件块类型
- 块内数据格式采用 YAML 列表
- 一个 block 内可包含多个事件

## 6. 与实体标注的衔接

### 6.1 实体引用

事件标注中的人物和地点应使用实体标注中的规范名，与实体索引保持一致。

### 6.2 段落引用

事件标注中的 `paragraphs` 字段应引用正文中的 Purple Number 段落编号，确保可验证性。

### 6.3 时间表达

事件标注中的 `time_text` 字段应与正文中的时间实体标注保持一致，`year` 字段应使用统一的上元纪年。

## 7. 事件抽取策略

### 7.1 混合抽取策略

第一阶段不要追求全自动复杂事件抽取，可采用混合方案：

1. **实体优先**：从段落中优先抽取已标注的人名、地名、时间
2. **结构识别**：按文章体裁和标题识别候选事件段
3. **人工补注**：对重点文章人工补一层事件索引 Markdown
4. **脚本生成**：脚本从“事件索引文件”生成标准 JSON

### 7.2 AI Agent 辅助抽取

1. **预处理**：确保正文已完成实体标注和段落编号
2. **AI 识别**：使用 AI Agent 分析文本，识别潜在事件
3. **人工审核**：审核 AI 生成的事件标注，修正错误和补充信息
4. **脚本校验**：运行校验脚本验证事件的合法性
5. **质量控制**：进入质量控制闭环

---

## 8. 质量控制机制

### 8.1 校验规则

事件标注应至少满足以下校验规则：

1. `id` 全局唯一
2. `type` 在白名单中
3. `article` 路径存在
4. `paragraphs` 中的段落编号在正文中存在
5. `year` 字段为有效数字
6. `people` 和 `places` 中的名称为规范名
7. 事件时间线逻辑自洽

### 8.2 质量控制闭环

建立质量控制闭环：

1. **抽取**：从标注文本中提取事件
2. **校验**：使用脚本验证事件的合法性
3. **反思**：人工审核和修正
4. **批改**：更新事件标注
5. **重建**：重新生成事件索引和时间线

---

## 9. 与技术路线方案的衔接

### 9.1 Phase 3 事件抽取与时间轴

- 事件标注是 Phase 3 的核心输入
- 事件 JSON 是时间线页面的数据源
- 事件索引是关系图谱的重要组成部分

### 9.2 Phase 4 关系图谱与质量校验

- 事件是关系图谱中的重要节点
- 事件参与关系（如 `participates_in`）是规则关系的重要来源
- 事件因果关系（如 `causes`）是推理关系的重要来源

### 9.3 Phase 5 图谱可视化页面

- 事件是时间线视图的核心元素
- 事件是章节事件轨道图/地铁图的站点
- 事件关系是图谱可视化的重要边

## 10. 标注流程建议

### 10.1 人工标注流程

1. **实体标注**：完成正文中的实体标注与段落编号
2. **事件识别**：识别文章中的关键事件
3. **事件标注**：补写事件文件，记录事件信息
4. **脚本校验**：运行校验脚本验证事件的合法性
5. **质量审查**：人工审核校验结果，修正错误
6. **重建索引**：重新生成事件索引和时间线数据

### 10.2 AI Agent 辅助标注流程

1. **预处理**：确保正文已完成实体标注和段落编号
2. **AI 生成**：使用 AI Agent 分析文本，生成事件标注建议
3. **人工审核**：审核 AI 生成的事件，修正错误和补充信息
4. **脚本校验**：运行校验脚本验证所有事件
5. **质量控制**：进入质量控制闭环

## 13. 最终要求

`NEXUS` 的事件标注，必须满足四个底线：

1. **可读**：人能看懂
2. **可抽取**：脚本能稳定解析
3. **可校验**：错误能自动发现
4. **可演进**：将来可生成时间线、关系网、图谱数据

如果某种写法只满足“看起来方便”，但不满足这四点，就不应进入正式规范。
